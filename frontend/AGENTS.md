## 3. 前端核心规则

### 3.1 Fluent 2 基线
前端可见 UI 以 **Fluent 2** 为唯一视觉基线。  
页面必须消费设计系统，而不是自行发明设计系统。

### 3.2 Fluent 2 文档优先
凡涉及以下内容，必须优先查阅 Fluent 2 官方文档：

- 布局
- 导航
- 表单
- 交互状态
- 排版
- 可访问性
- 设计 token
- 后台管理界面模式

### 3.3 页面职责
页面不直接承担：

- 复杂业务编排
- 权限最终判定
- 高风险动作的真实执行逻辑
- 远程服务控制逻辑
- 审计写入逻辑

页面负责：

- 展示状态
- 组织交互
- 调用 API
- 呈现反馈
- 正确表达权限、风险与结果

### 3.4 分层
推荐结构：

- `layouts/`
- `pages/`
- `components/`
- `stores/`
- `api/`
- `composables/`
- `types/`
- `utils/`
- `styles/`

### 3.5 权限展示原则
权限最终以后端为准。  
前端只做体验优化，不做安全兜底。

要求：

- 根据后端返回的权限码控制菜单、按钮、页面入口
- 不硬编码“某角色一定能做什么”
- 不用隐藏按钮代替真正权限控制
- 403 / 无权限状态要明确展示
- 对受限能力要告诉用户原因，而不是静默消失

### 3.6 高风险交互原则
涉及以下能力时：

- 删除
- 封禁
- 提权
- SSH 会话创建
- FRP 配置变更
- 邮件发送
- 工作区销毁
- Minecraft 启停 / 重启
- 系统配置修改

必须尽量提供：

- 明确确认弹窗
- 影响说明
- 风险提示
- 加载与失败状态
- 成功后的结果反馈
- 后端返回的审计编号或状态信息（若有）

## 4. 前端代码规范

### 4.1 规范来源
优先遵循：

1. Vue 3 官方规范
2. Fluent 2 官方设计规范
3. TypeScript 官方规范
4. 仓库既有约定
5. 通用工程约定

### 4.2 类型与命名
严格要求：

- 禁止 `var`
- 尽量不使用 `any`
- 优先 `const` / `let`
- 类型定义要有业务语义
- 禁止无语义命名，如 `tmp`、`data`、`obj`、`res`
- 组件名、store 名、API 名必须体现职责

### 4.3 组件与设计系统
- 共享视觉元素优先沉淀为设计系统组件
- 页面层不要反复硬编码颜色、间距、圆角、阴影、状态色
- 导航、表单、反馈、卡片、表格、列表保持 Fluent 2 风格一致
- 不同模块不要出现多套视觉语言

### 4.4 组件边界
- 展示组件与业务组件分离
- 布局组件与页面组件分离
- 表单状态与展示分离
- 风险确认组件与实际业务请求分离
- 文件展示组件与文件权限逻辑分离

### 4.5 未完成实现
默认不保留：

- 空页面
- 假按钮
- 未接线弹窗
- 假数据成功态
- 未处理错误态
- 无权限却仍可点击的危险操作

若确有必要，使用 `TODO:` / `FIXME:` 并说明原因与影响。

## 5. 页面与模块专项要求

### 5.1 布局基线
默认布局：

- `PublicLayout`
- `DashboardLayout`
- `AdminLayout`

涉及后台与工具平台页面时，优先保持信息密度、层级和操作反馈的一致性。

### 5.2 模块页面基线
以下模块页面必须特别注意权限与状态表达：

- 用户与权限管理
- 存储与临时传输
- SSH
- FRP
- 邮件
- 远程开发
- Minecraft
- 系统配置
- 审计日志

### 5.3 表单与列表
- 表单要有完整校验与错误提示
- 列表页要有分页、筛选、空态、加载态
- 高风险操作入口不得与普通操作视觉等权
- 批量操作必须清楚表达作用对象和范围

### 5.4 API 状态映射
状态与 API 映射逻辑不要散落在模板中。  
请求、错误、重试、禁用态、成功态统一放在组合式逻辑或 store 中管理。
