# ADR-002: IAM 权限模型决策

> 状态: **已采纳**  
> 日期: 2025

---

## 背景

QWeb 包含 17 个功能模块，涉及内容发布、文件管理、SSH 网关、服务器控制等多种敏感程度不同的操作。需要一套灵活的权限系统来控制不同用户对不同功能的访问。

## 决策

### 采用三层权限模型

```text
第一层：RBAC（基于角色的权限控制）
第二层：用户级覆盖（针对个别用户的 allow/deny）
第三层：资源级策略（后续扩展）
```

### 预置角色

| 角色 | 定位 |
|------|------|
| owner | 系统所有者，跳过所有检查 |
| admin | 站点管理，管理大部分业务 |
| trusted | 受信任用户（朋友），额外功能 |
| user | 注册用户 |
| guest | 匿名访客 |

### 权限判断顺序

```text
1. owner → 直接通过
2. 检查 is_disabled → 如果禁用，拒绝
3. 检查 deny 覆盖 → 如果匹配，拒绝
4. 检查 allow 覆盖 → 如果匹配，通过
5. 检查角色权限 → 是否包含所需权限码
6. 检查资源策略 → 精细控制（后续）
7. 默认拒绝
```

### 权限码命名规范

```text
{module}.{resource}.{action}

示例:
blog.post.create
blog.post.update
iam.role.assign
ssh_gateway.session.create
```

## 理由

- **为什么不用纯 RBAC？**  
  纯角色权限无法处理"某个用户需要额外开放/关闭某个功能"的场景。用户级覆盖解决了这个问题。

- **为什么不用 ABAC？**  
  属性策略过于复杂，当前规模不需要。三层模型在简单和灵活之间取得平衡。

- **为什么 owner 跳过检查？**  
  个人项目，owner 是站长本人，无需限制。减少维护负担。

## 替代方案

| 方案 | 评估 | 否决原因 |
|------|------|----------|
| 纯 RBAC | 简单 | 缺少用户级精细调整能力 |
| Django Groups + Permissions | 内置 | Django 权限粒度不够，无覆盖概念 |
| ABAC | 灵活 | 实现和维护复杂度过高 |
| 第三方服务 (Casbin) | 成熟 | 引入外部依赖，个人项目偏重 |

## 影响

- 所有模块必须通过 IAM PermissionResolver 做权限检查
- 禁止在业务代码中硬编码角色名判断
- 前端必须在登录后拉取权限列表，用于 UI 元素显示控制
- 审计模块必须记录权限拒绝事件
