# 模块边界与职责划分

> 17 个 Django App 的职责定义、层级归属与协作约束

---

## 1. 三层架构

```
┌─────────────────────────────────────────────────────┐
│  领域层 Domain                                       │
│  accounts · iam · blog · wiki · storage ·            │
│  temp_transfer · ssh_gateway · frp_manager ·         │
│  mc_manager · mail_center · dev_workspace ·          │
│  network_tools                                       │
├─────────────────────────────────────────────────────┤
│  平台层 Platform                                     │
│  audit · notifications · system_config ·             │
│  media_center                                        │
├─────────────────────────────────────────────────────┤
│  基础层 Base                                         │
│  core · common · config                              │
└─────────────────────────────────────────────────────┘
```

- **领域层**：所有业务功能，直接面向用户
- **平台层**：为领域层提供横切能力（审计、通知、配置、媒体）
- **基础层**：基类、公共工具、全局配置

---

## 2. 基础层模块

### 2.1 `core`

| 项目 | 说明 |
|------|------|
| 职责 | 全局基础配置、通用基类、公共工具函数、响应格式、异常处理、健康检查 |
| 包含 | BaseModel（时间戳、软删除、操作者字段）、公共 Mixin、全局常量 |
| 依赖 | 无业务模块依赖 |
| 被依赖 | 所有模块 |

### 2.2 `common`

| 项目 | 说明 |
|------|------|
| 职责 | 公共数据库工具、异常类、统一响应、分页、中间件、公共序列化器、公共权限基类 |
| 目录 | `db/` `exceptions/` `responses/` `security/` `pagination/` `middleware/` `permissions/` `serializers/` |
| 依赖 | core |
| 被依赖 | 所有模块 |

---

## 3. 平台层模块

### 3.1 `audit`

| 项目 | 说明 |
|------|------|
| 职责 | 审计日志、登录日志、权限变更日志、系统操作日志、高危操作记录 |
| 写入方 | 所有领域模块通过统一接口写入审计 |
| 约束 | 审计记录与业务动作同步但不耦死；高风险模块必须记录审计 |

### 3.2 `notifications`

| 项目 | 说明 |
|------|------|
| 职责 | 站内通知、邮件通知、系统消息、异步消息派发 |
| 约束 | 通知发送走异步队列（Celery），不阻塞业务请求 |

### 3.3 `system_config`

| 项目 | 说明 |
|------|------|
| 职责 | 站点配置、开关项、模块启停、注册策略、上传限制、邮件策略、黑白名单 |
| 约束 | 模块开关统一从此模块读取；配置解析不散落到各业务模块 |

### 3.4 `media_center`

| 项目 | 说明 |
|------|------|
| 职责 | 图片、附件、封面管理、统一媒体引用 |
| 约束 | blog/wiki/storage 共用媒体能力；不各自重复实现上传逻辑 |

## 4. 领域层模块

### 4.1 `accounts`

| 项目 | 说明 |
|------|------|
| 职责 | 用户注册、登录、用户资料、邮箱验证、密码重置、用户状态管理 |
| 包含 | 自定义 User 模型、UserProfile、注册/认证 API、安全设置 |
| 与 IAM 关系 | accounts 管身份，iam 管授权；accounts 不做权限判定 |

### 4.2 `iam`

| 项目 | 说明 |
|------|------|
| 职责 | 角色管理、权限定义、角色权限绑定、用户权限覆盖、权限校验服务、资源策略预留 |
| 核心地位 | **最重要的独立模块**，所有模块的权限校验统一走 IAM |
| 约束 | 禁止在业务代码中硬编码角色名判断；禁止绕过 IAM 直接放行敏感操作 |

### 4.3 `blog`

| 项目 | 说明 |
|------|------|
| 职责 | 博客文章、分类、标签、评论（可选）、草稿/发布、可见性控制 |
| 依赖 | media_center（封面/图片）、iam（权限）、audit（操作审计） |

### 4.4 `wiki`

| 项目 | 说明 |
|------|------|
| 职责 | Wiki 页面、版本历史、空间/目录、编辑权限、页面链接关系 |
| 依赖 | media_center（附件）、iam（空间级权限）、audit |

### 4.5 `storage`

| 项目 | 说明 |
|------|------|
| 职责 | 小型云盘、文件上传/下载、文件夹管理、分享链接、配额控制、文件权限 |
| 与 temp_transfer 区别 | storage 是**长期存储**，temp_transfer 是**临时流转** |

### 4.6 `temp_transfer`

| 项目 | 说明 |
|------|------|
| 职责 | 临时文件传输、临时上传口、临时分享、过期清理、一次性链接 |
| 约束 | 独立于 storage；过期清理走 Celery Beat 定时任务 |

### 4.7 `network_tools`

| 项目 | 说明 |
|------|------|
| 职责 | 测速、网络诊断、Ping/Traceroute（需谨慎）、端口检查、下载测速历史 |
| 风险等级 | **高风险** — 需限制调用频率与范围 |

### 4.8 `frp_manager`

| 项目 | 说明 |
|------|------|
| 职责 | FRP 配置管理、隧道列表、状态查看、配置下发、使用审计 |
| 风险等级 | **高风险** — 涉及网络隧道与端口暴露 |

### 4.9 `ssh_gateway`

| 项目 | 说明 |
|------|------|
| 职责 | SSH 连接配置、目标设备管理、Web 终端接入、会话权限控制、命令审计、会话记录 |
| 风险等级 | **高风险** — 与 IAM、Audit 深度集成 |
| 实时通道 | WebSocket `/ws/ssh/sessions/{id}` |

### 4.10 `mc_manager`

| 项目 | 说明 |
|------|------|
| 职责 | Minecraft 服务器实例管理、开关机、状态监控、玩家信息、日志查看、备份任务、白名单/配置管理 |
| 风险等级 | **高风险** — 涉及服务控制 |
| 实时通道 | WebSocket `/ws/mc/servers/{id}/console` |

### 4.11 `mail_center`

| 项目 | 说明 |
|------|------|
| 职责 | 邮件账号配置、发件记录、SMTP/IMAP 接入配置、通知模板、系统邮件派发 |
| 风险等级 | **高风险** — 涉及 SMTP 凭据与发送能力 |

### 4.12 `dev_workspace`

| 项目 | 说明 |
|------|------|
| 职责 | 远程开发入口、容器/工作区管理、Web IDE 跳转、代码服务接入、开发会话状态、权限控制 |
| 风险等级 | **高风险** — 涉及远程代码执行环境 |

## 5. 模块协作约束

### 5.1 跨模块调用规则

```
✅ 允许：通过对方暴露的 service / selector 调用
✅ 允许：领域层 → 平台层（写审计、读配置、发通知、引用媒体）
✅ 允许：所有模块 → 基础层

❌ 禁止：跨模块直接 import 对方内部 model 逻辑
❌ 禁止：平台层 → 领域层（平台层不反向依赖业务）
❌ 禁止：绕过 IAM 自行实现权限判定
❌ 禁止：绕过 Audit 静默执行高危操作
❌ 禁止：绕过 system_config 硬编码模块开关
```

### 5.2 统一能力归口

| 能力 | 归口模块 | 说明 |
|------|----------|------|
| 权限校验 | `iam` | 所有模块走统一 `PermissionResolver` |
| 审计记录 | `audit` | 高危操作统一写日志 |
| 系统配置 | `system_config` | 模块开关、策略配置统一读取 |
| 文件上传 | `media_center` | 封面、附件、图片不各自实现 |
| 异步任务 | Celery | 长耗时任务不塞进同步请求 |
| 实时通道 | Channels | SSH、MC 控制台统一走 WebSocket 规范 |
| 通知派发 | `notifications` | 站内通知、邮件通知统一入口 |

### 5.3 高风险模块清单

以下模块默认视为高风险，涉及时需额外审慎：

- `ssh_gateway` — 远程主机控制
- `frp_manager` — 网络隧道
- `mail_center` — SMTP 凭据
- `dev_workspace` — 远程执行环境
- `mc_manager` — 服务控制
- `network_tools` — 网络诊断
- 任何涉及系统配置、权限、密钥、删除、停机、重启的功能

## 6. 扩展性设计

### 6.1 新增模块接入流程

新增业务模块（如 `git_service`、`vpn_manager`、`rss_reader`）时：

1. 在 `apps/` 下创建新 app，遵循标准内部结构
2. 在 `iam` 中注册模块权限码
3. 在 `system_config` 中注册模块开关
4. 在 `audit` 中注册高危操作类型（如有）
5. 文件上传能力走 `media_center`
6. 异步任务走 Celery
7. 实时通道走 Channels 规范
8. 在 `docs/` 中更新相关文档

### 6.2 模块隔离保证

- 每个模块只暴露自己的 service 和 API
- 模块停用时不影响其他模块正常运行
- 数据库表按模块前缀组织
- 测试按模块独立运行
