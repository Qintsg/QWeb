# 一、总体架构设计目标

一个**个人基础设施门户 + 内容平台 + 工具中台**，所以架构上建议采用：

- **前后端分离**
- **领域模块化**
- **统一身份与权限中心（IAM）**
- **统一资源接入层**
- **统一审计与配置中心**
- **插件化业务模块**

建议整体采用：

- 后端：**Django + Django REST Framework**
- 前端：**Vue 3 + TypeScript + Vite**
- 数据库：**PostgreSQL**
- 缓存/队列：**Redis**
- 异步任务：**Celery**
- 实时通信：**Django Channels**
- 对象存储：**本地文件系统 / MinIO（后期）**
- 反向代理：**Nginx / Caddy**
- 容器化：**Docker Compose（初期）**
- 权限模型：**RBAC + 用户级覆盖 + 资源级策略扩展**
- UI 风格：**Fluent 2 Design System**

# 三、权限模型设计（重点）

这是整个系统最关键的部分。
你的要求是：

- 用户可自行注册
- 管理员可自由调整访问限制
- 有五个用户组：`owner`、`admin`、`trusted`、`user`、`guest`
- 需要有**预置、可修改**的权限
- 对个人权限调整是在**用户组权限基础上进行覆盖**
- 需要**充分可扩展**、**高度模块化**

这意味着你不能只用 Django 默认 Group 做完事，而应该做一个**独立 IAM 子系统**。

------

## 3.1 建议的权限模型

采用三层模型：

### 1）权限定义层

定义系统中所有可授权能力，例如：

- `site.view_dashboard`
- `blog.post.create`
- `blog.post.publish`
- `wiki.page.edit`
- `storage.file.upload`
- `storage.file.delete`
- `ssh.session.create`
- `ssh.session.execute`
- `mc.server.start`
- `mc.server.stop`
- `mail.account.use`
- `admin.user.manage`
- `admin.role.manage`
- `admin.permission.manage`

权限按**模块 + 资源 + 动作**命名。

------

### 2）角色/用户组层

系统预置五个角色：

- `owner`
- `admin`
- `trusted`
- `user`
- `guest`

说明：

- 这是**系统预置角色**
- 可以修改它们的权限集合
- 可以额外新增自定义角色（建议支持）
- 角色可以绑定多个权限

------

### 3）用户覆盖层

用户最终权限 = **角色权限** + **用户覆盖**

覆盖分两类：

- `allow`：额外授予
- `deny`：显式拒绝

例如：

- `trusted` 默认不能访问 SSH
- 某个用户额外加 `ssh.session.create` -> allow
- `admin` 默认可访问 Wiki 管理
- 某个 admin 被限制 `wiki.page.delete` -> deny

------

## 3.2 权限判定顺序

建议统一规则：

```text
1. owner 直接通过（可配置为系统保底最高权限）
2. 检查用户是否被禁用/锁定
3. 检查用户显式 deny
4. 检查用户显式 allow
5. 检查角色权限
6. 检查资源级策略（若启用）
7. 默认拒绝
```

这样逻辑清晰，也容易调试。

------

## 3.3 资源级权限扩展

为了未来可扩展，建议预留资源级授权能力，例如：

- 某篇博客仅 trusted 可见
- 某个 Wiki 空间仅 admin 与 owner 可编辑
- 某个文件夹仅自己和 trusted 可访问
- 某台 Minecraft 服务仅 owner/admin 可操作

因此需要在 IAM 中加入可选能力：

- `resource_type`
- `resource_id`
- `subject_type`（user / role）
- `effect`（allow / deny）

第一版可以先不全面启用，但**模型一定要预留**。

------

## 3.4 推荐数据模型

建议建立这些表：

### 基础身份

- `users`
- `user_profiles`

### 权限系统

- `permissions`
- `roles`
- `role_permissions`
- `user_roles`
- `user_permission_overrides`
- `resource_policies`（可选，但建议先建）

### 审计

------

## 3.5 推荐字段示意

## Permission

- `id`
- `code`：如 `blog.post.create`
- `name`
- `module`
- `resource`
- `action`
- `description`
- `is_builtin`
- `is_active`

## Role

- `id`
- `code`：`owner/admin/trusted/user/guest`
- `name`
- `description`
- `is_builtin`
- `is_system`
- `priority`

## RolePermission

- `role_id`
- `permission_id`

## UserRole

- `user_id`
- `role_id`
- `expires_at`（可选，支持临时授权）

## UserPermissionOverride

- `user_id`
- `permission_id`
- `effect`：`allow/deny`
- `reason`
- `expires_at`

## ResourcePolicy

- `subject_type`：`user/role`
- `subject_id`
- `resource_type`
- `resource_id`
- `permission_id`
- `effect`

------

## 3.6 系统预置角色建议

## owner

拥有系统级全部权限，建议始终保留至少一个 owner。

## admin

站点管理、用户管理、内容管理、工具管理，但不一定拥有 owner 保底权限。

## trusted

适合熟人，可访问受信资源，如临时文件分享、特定 Wiki、受限服务。

## user

普通注册用户，可浏览公开内容、使用受控基础功能。

## guest

未登录或访客身份，主要访问公开页面。

# 七、API 设计

建议统一风格：

- 前缀：`/api/v1/`
- 认证：JWT Bearer Token
- 响应规范统一
- 所有模块都按 REST 风格组织
- 对实时能力用 WebSocket

------

## 7.1 统一响应格式

成功：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

失败：

```json
{
  "code": 4001,
  "message": "permission denied",
  "errors": {
    "permission": ["blog.post.publish"]
  }
}
```

# 十、后端 app 内部推荐结构

每个 Django app 内建议尽量统一结构，不要全都塞一个 `views.py`。

例如 `backend/apps/iam/`：

```text
iam/
├─ apps.py
├─ admin.py
├─ urls.py
├─ models/
│  ├─ __init__.py
│  ├─ permission.py
│  ├─ role.py
│  ├─ user_role.py
│  ├─ override.py
│  └─ resource_policy.py
├─ api/
│  ├─ serializers/
│  ├─ views/
│  └─ filters.py
├─ services/
│  ├─ permission_resolver.py
│  ├─ role_service.py
│  ├─ override_service.py
│  └─ policy_service.py
├─ permissions/
│  ├─ decorators.py
│  ├─ drf_permissions.py
│  └─ checker.py
├─ signals/
├─ selectors/
├─ tests/
└─ migrations/
```

推荐模式：

- `models/`：模型定义
- `api/`：序列化与视图
- `services/`：业务逻辑
- `selectors/`：查询逻辑
- `permissions/`：权限判定
- `tests/`：模块内测试

这样比传统 Django 大文件更适合大项目。

# 十一、权限实现建议

## 11.1 不建议只依赖 Django 自带 Group 的原因

因为它不够表达：

- 用户级 allow/deny 覆盖
- 资源级策略
- 角色模板可编辑
- 系统预置角色与业务权限绑定
- 权限调试与审计

所以建议：

- Django 自带 auth 仅做基础认证支持
- 真正授权放在 `iam` 模块里

------

## 11.2 推荐权限检查方式

后端写统一装饰器 / DRF Permission Class：

```python
@require_permission("blog.post.publish")
def publish_post(...):
    ...
```

或者 DRF 里：

```python
class BlogPublishPermission(BasePermission):
    required_permission = "blog.post.publish"
```

内部走统一 `PermissionResolver`：

```text
resolve_user_permissions(user, resource=None)
has_permission(user, permission_code, resource=None)
```

------

## 11.3 前端权限控制

前端不负责安全，只负责体验优化。

建议前端在登录后拉取：

- 用户角色
- 用户最终权限列表
- 模块可见性配置

然后：

- 控制菜单显示
- 控制按钮禁用
- 控制路由访问体验

但真正校验必须以后端为准。

# 十二、可扩展性关键设计

为了“高度模块化”，建议坚持下面几条：

## 1）每个模块只暴露自己的 service 和 API

不要跨模块直接 import 对方内部模型逻辑太深。

## 2）权限统一从 IAM 走

不要每个模块私自搞一套权限。

## 3）审计统一从 Audit 走

所有高危操作统一写日志。

## 4）模块开关统一从 system_config 管

以后可以停用某模块而不影响整体。

## 5）文件能力统一从 media/storage 抽象

博客封面、Wiki 附件、临时传输不要各写一份上传逻辑。

## 6）实时通道统一约定

SSH、MC 控制台、任务状态都走 Channels/WebSocket 规范。

# 十四、我对这套方案的最终建议

这类站点最容易失败的点不是功能少，而是**权限混乱、模块耦合、后期扩展困难**。
所以你这一版最应该先立住的是：

- **自定义 User + IAM 中心**
- **统一模块边界**
- **统一 API 风格**
- **统一审计机制**
- **统一配置中心**
- **统一前端布局与 Fluent 2 设计 token**
