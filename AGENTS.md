# AGENTS.md

## 0. 唯一人工权限开关

> 仓库内所有 AI 权限只看这一个字段。  
> 子目录 `AGENTS.md` 只允许引用和解释，不得重定义。

**AI_PERMISSION_LEVEL: DANGEROUS**

可选值：

- `LOCKED`
- `STANDARD`
- `PRIVILEGED`
- `DANGEROUS`

权限摘要：

- `LOCKED`：只读检查、常规改码、常规测试、常规文档更新、常规 commit
- `STANDARD`：中等范围重构、受控脚本、受控本地调试、受控前后端联调
- `PRIVILEGED`：敏感配置改动、受控数据库读写、受控基础设施配置修复、受控冲突处理
- `DANGEROUS`：仅在明确任务下执行高风险数据库、服务、结构、清理、冲突、历史改写动作

共同原则：

- 最小必要
- 优先可回滚
- 优先代码层修复
- 优先受控验证
- 涉及高权限动作必须说明理由、影响范围、验证结果

## 3. 通用工作原则

### 3.1 先理解，再修改
先阅读：

- 相邻文件
- 当前模块目录
- 现有分层与服务边界
- API 契约
- 权限与角色实现
- 审计与配置实现
- 测试与文档
- 与当前模块直接相关的前后端代码

### 3.2 做最小且正确的改动
优先：

- 最小影响面
- 最低回滚成本
- 与既有架构一致
- 容易审查
- 容易测试
- 容易追责

### 3.3 不混淆职责
必须保持分层清晰：

- Django 事务型业务与基础设施控制分离
- 认证与授权分离
- 角色权限与用户覆盖分离
- 页面展示与业务编排分离
- 配置读取与执行动作分离
- 审计记录与业务动作同步但不耦死
- 长耗时任务与请求响应分离

### 3.4 不臆造需求
只实现：

- 用户明确要求的内容
- 完成任务所必需的配套修改
- 代码上下文中可以直接推断的内容

禁止擅自增加：

- 未被请求的新业务模块
- 未接线的空壳服务
- 没有落地场景的抽象层
- 与当前任务无关的大重构
- 无依据的“未来预留”数据库字段和接口

### 3.5 未完成实现
默认要求是：

- 可运行
- 可验证
- 可交付
- 可审查

除非确有必要，不保留：

- 只有名字的空函数
- `pass`
- 假成功返回
- 未接线按钮
- 未接线 API
- 未接线权限码
- 未接线审计逻辑

若确有必要，使用：

- `TODO:`
- `FIXME:`

并写明：

- 原因
- 影响范围
- 后续动作
- 风险说明

### 3.6 最终表达
最终代码、注释、文档、提交说明中，只保留稳定、正向、面向结果的内容。  
不得保留过程性残留表达。

## 4. 业务与架构事实

### 4.1 统一身份与权限模型
本项目的权限模型是：

- 用户组：`owner`、`admin`、`trusted`、`user`、`guest`
- 预置角色可修改
- 用户最终权限 = 用户组权限 + 用户级覆盖
- 覆盖分为 `allow` 与 `deny`
- 后续可扩展到资源级策略

禁止：

- 在业务代码中到处硬编码角色名判断
- 绕过 IAM 直接放行敏感操作
- 只在前端做权限控制而后端不校验
- 将“是否管理员”写成唯一权限模型

### 4.2 模块职责基线
默认职责如下：

- `accounts`：注册、登录、资料、安全设置
- `iam`：角色、权限、覆盖、策略
- `audit`：登录日志、操作日志、高危动作审计
- `system_config`：站点配置、模块开关、策略配置
- `blog` / `wiki`：内容管理
- `storage` / `temp_transfer`：长期存储与临时传输
- `network_tools`：测速与网络诊断
- `frp_manager`：隧道配置与状态
- `ssh_gateway`：主机、会话、终端、命令审计
- `mc_manager`：Minecraft 服务控制与状态
- `mail_center`：邮件账号、模板、发件与记录
- `dev_workspace`：远程开发工作区与接入信息

### 4.3 高风险模块
以下模块默认视为高风险：

- `ssh_gateway`
- `frp_manager`
- `mail_center`
- `dev_workspace`
- `mc_manager`
- `network_tools`
- 任何涉及系统配置、权限、密钥、删除、停机、重启的功能

对这些模块：

- 没有明确任务，不做破坏性动作
- 优先配置校验、模拟检查、只读检查
- 需要记录影响范围
- 必须保证可审计
- 必须避免把敏感信息写入日志、前端、测试快照或文档

### 4.4 数据与存储职责
默认职责如下：

- **PostgreSQL**：用户、角色、权限、审计、内容、文件元数据、服务元数据、任务记录、系统配置
- **Redis**：缓存、队列、节流、锁、异步任务状态
- **文件存储**：上传文件、媒体资源、备份产物、临时传输对象
- **`.env` / Secret 管理**：凭据、令牌、SMTP、SSH、第三方密钥

未经明确要求，不要混淆这些职责。

## 5. Git 工作流

### 5.1 开始任务前
固定流程：

1. `git status`
2. 检查未提交内容
3. 若有未提交内容：
   - 与当前任务相关且已稳定：先 commit
   - 与当前任务无关：
     - `LOCKED` / `STANDARD`：请示
     - `PRIVILEGED` / `DANGEROUS`：可做保护性 commit
4. 状态安全后执行 `git pull --rebase`
5. pull 产生冲突时按权限规则处理
6. pull 完成后再开始改动

### 5.2 提交节奏
每完成一个独立可审查块，应 commit 一次。  
典型块包括：

- 一个 API 修复
- 一个权限逻辑修复
- 一个高风险模块的受控修复
- 一个页面完整调整
- 一次文档同步更新
- 一组测试修复

### 5.3 commit 规范
格式：

```text
type(scope): 中文摘要
````

可用 `type`：

* `feat`
* `fix`
* `refactor`
* `docs`
* `style`
* `test`
* `perf`
* `build`
* `ci`
* `chore`
* `revert`

示例：

```text
feat(iam): 新增用户权限覆盖解析逻辑
fix(storage): 修复分享链接过期判断
refactor(ssh_gateway): 收敛会话创建服务边界
docs(architecture): 更新权限模型说明
test(blog): 补充文章发布权限测试
```

### 5.4 默认禁止的 Git 动作

除非用户明确要求，否则不主动执行：

* `git push`
* 切换分支
* 新建分支
* 提交 PR
* 改写远端历史
* 删除 tag

### 5.5 冲突处理

* `LOCKED`：请示
* `STANDARD`：仅处理明显无歧义的小冲突；复杂冲突请示
* `PRIVILEGED`：可自主处理冲突，处理后必须验证并说明依据
* `DANGEROUS`：可处理高复杂度冲突，但必须记录影响与风险

## 6. 文档规则

### 6.1 文档位置

说明文档统一放在：

* `docs/`

重点文档优先同步到相关位置：

* `docs/architecture/system-overview.md`
* `docs/architecture/iam-design.md`
* `docs/architecture/api-design.md`
* `docs/backend/django-apps.md`
* `docs/backend/models.md`
* `docs/frontend/routing.md`
* `docs/frontend/design-system.md`
* `docs/api/openapi.yaml`
* `docs/decisions/*.md`

### 6.2 完成工作后必须同步更新文档

只要变更影响：

* API 契约
* 权限模型
* 角色与覆盖逻辑
* 数据结构
* 审计策略
* 配置项
* 部署 / 运行方式
* 前端关键交互
* 页面路由
* 高风险模块行为

就必须同步更新 `docs/` 下相关文档。

### 6.3 更新原则

更新文档时：

* 保留原有总体结构
* 只在相关章节增补或修订
* 保持标题层级与既有风格
* 不随意重写整份文档

### 6.4 版本号

除非用户明确要求或任务本身就是发版，不修改版本号。

### 6.5 目录新增限制

`frontend/` 与 `backend/` 下默认不新增说明型文档。
允许新增的必要文件仅限：

* `.env.example`
* 依赖或工具链必须存在的标准文件
* 构建、运行、部署、测试所需配置模板

## 8. 全仓库代码规范

### 8.1 通用要求

除子目录文件另有规定外，代码规范遵循：

* 官方标准文档
* 仓库既有约定
* 通用工程实践

### 8.2 类型与命名

统一要求：

* 命名必须有业务语义
* 不使用 `var`、`tmp`、`foo`、`bar`、`data`、`obj` 这类弱语义名称
* 优先显式类型与清晰边界
* 禁止用注释弥补糟糕命名

### 8.3 注释要求

注释必须解释：

* 意图
* 约束
* 边界
* 风险
* 为什么这样做

禁止大量重复代码表面含义的无效注释。

### 8.4 安全要求

禁止：

* 在代码、日志、截图、测试数据、文档中泄露真实密钥
* 把敏感配置写死在源码中
* 在高风险模块中默认放开权限
* 删除或绕过审计
* 为了方便调试而永久关闭安全检查

## 10. 完成定义

一个任务完成时必须同时满足：

* 用户要求已实现
* 规则已遵守
* 代码可运行、可验证、可审查
* 权限、审计、配置、接口未被意外破坏
* 高风险模块未被越权修改
* 文档已同步
* 当前块内容已 commit
* 最终内容中无过程性残留表达
